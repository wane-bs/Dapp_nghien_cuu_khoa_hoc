# Báo Cáo Nghiên Cứu Chuyên Sâu: Tối Ưu Hóa Và Tái Cấu Trúc Dự Án VinaLib-Vault Cho Vận Hành Localhost Zero-Cost

## 1. Tổng Quan Điều Hành: Chiến Lược Chuyển Đổi Sang Môi Trường Giả Lập

Sự phát triển của các hệ thống Cyber-Physical (Thực-Ảo) phức tạp như VinaLib-Vault (dựa trên kiến trúc tham chiếu VinaStay-Invest) đòi hỏi sự tích hợp chặt chẽ giữa Hợp đồng thông minh (Smart Contracts), thiết bị IoT vật lý, và các khung pháp lý số.1 Tuy nhiên, trong giai đoạn phát triển và kiểm thử (Development & Testing), sự phụ thuộc vào các dịch vụ bên thứ ba (Third-party services) như Tuya Cloud, Chainlink DON (Decentralized Oracle Network), FPT.eContract và hệ thống ngân hàng qua Casso tạo ra những rào cản đáng kể về chi phí, độ trễ và tính phức tạp. Yêu cầu đặt ra là phải tái cấu trúc hệ thống để vận hành toàn diện trên môi trường cục bộ (localhost) với chi phí vận hành bằng 0 (Zero-Cost), đồng thời đảm bảo tính trung thực của logic nghiệp vụ.

Báo cáo này trình bày một bản thiết kế kiến trúc chi tiết nhằm thay thế toàn bộ các phụ thuộc bên ngoài bằng các giải pháp Mocking (Giả lập đối tượng) và Simulation (Mô phỏng hành vi) tiên tiến. Thay vì chỉ đơn thuần là ngắt kết nối mạng, chúng ta xây dựng một "Digital Twin" (Bản sao số) của môi trường sản xuất ngay trên máy trạm của lập trình viên. Chiến lược này chuyển dịch hệ thống từ mô hình "Trust-Minimized" (Giảm thiểu niềm tin - yêu cầu bắt buộc cho Mainnet) sang mô hình "Trust-Simulated" (Giả lập niềm tin - tối ưu cho Devnet), cho phép kiểm thử các kịch bản biên (edge cases) mà không tốn phí gas hay phí API.

Việc tái cấu trúc dựa trên nguyên lý Dependency Inversion (Đảo ngược sự phụ thuộc), nơi các logic cốt lõi không giao tiếp trực tiếp với các SDK của bên thứ ba, mà thông qua các lớp Trừu tượng (Abstraction Layers). Trong môi trường Local, các lớp này sẽ được tiêm (inject) các Mock Adapter thay vì Real Adapter. Phân tích sâu dưới đây sẽ đi vào chi tiết kỹ thuật cho từng lớp trong kiến trúc 6 lớp của VinaLib-Vault 1, cung cấp các lược đồ JSON, các đoạn mã giả lập Hardhat, và các chiến lược định tuyến lại API để đạt được mục tiêu vận hành zero-cost.

## 2. Lớp 1: Giả Lập Hạ Tầng Pháp Lý Số (FPT.eContract)

Trong kiến trúc VinaLib-Vault, lớp pháp lý đóng vai trò neo giữ giá trị của token vào một văn bản pháp lý có hiệu lực thực tế, sử dụng dịch vụ FPT.eContract để ký số.1 Việc gọi API thật của FPT mỗi khi chạy test suite là bất khả thi do chi phí tạo hợp đồng và yêu cầu chữ ký số vật lý (USB Token/HSM). Do đó, giải pháp Mocking ở đây tập trung vào việc mô phỏng quy trình "Tạo - Ký - Xác thực" của API FPT.

### 2.1 Phân Tích Giao Thức FPT.eContract Và Chiến Lược Mocking

Quy trình thực tế yêu cầu gửi metadata (thông tin nhà đầu tư, tài sản) lên FPT, nhận về file PDF, thực hiện ký số, và sau đó xác thực lại file đã ký.1 Để giả lập quy trình này trên localhost, chúng ta cần xây dựng một Local Mock Server (sử dụng Node.js/Express hoặc json-server) để chặn các request này.

Theo tài liệu tham khảo về FPT.AI và các cấu trúc API tương đương 2, Mock Server cần phản hồi các cấu trúc JSON chuẩn xác để Backend của VinaLib-Vault không nhận ra sự khác biệt.

#### Chiến lược "Deterministic Hashing" (Băm Tất Định)

Vấn đề cốt lõi khi giả lập chữ ký số là Smart Contract cần một mã băm (hash) của tài liệu để mint token.1 Trong môi trường giả lập, thay vì tạo ra một file PDF thực sự (tốn tài nguyên CPU), Mock Server sẽ tạo ra một chuỗi hash giả định nhưng nhất quán dựa trên đầu vào.

Quy trình Giả lập:

Request: Backend gửi POST /api/v1/contract/create với payload { "investorId": "INV-001", "assetId": "VILLA-88" }.

Mock Logic: Server giả lập sẽ nối chuỗi INV-001 + VILLA-88 + SECRET_SALT và thực hiện băm SHA-256. Kết quả này sẽ đóng vai trò là contract_hash.

Response: Trả về JSON với trạng thái thành công và mã hash vừa tạo.

### 2.2 Đặc Tả Kỹ Thuật Mock API (FPT Layer)

Dưới đây là cấu trúc dữ liệu JSON cần thiết lập cho Mock Server để thay thế dịch vụ thật:

Endpoint Giả Lập: POST /electronic-sign/create

Mục đích: Mô phỏng việc khởi tạo luồng ký kết hợp đồng.

Payload Request (Từ Backend VinaLib):

JSON

{  "template_code": "SPA_VILLA_TEMPLATE",  "signers": [    { "name": "Nguyen Van A", "email": "a@test.local", "id_card": "0123456789" }  ],  "metadata": { "token_id": "101", "price": "5000000000" }}

Payload Response (Từ Mock Server):

Dựa trên cấu trúc phản hồi mẫu 3, phản hồi giả lập cần bao gồm các trường định danh và trạng thái:

JSON

{  "code": 0,  "message": "Success",  "data": {    "document_id": "mock-doc-uuid-123456789",    "access_code": "ac-9999",    "status": "DRAFT",    "signing_url": "http://localhost:3000/mock-sign-ui/mock-doc-uuid-123456789",    "created_at": "2025-12-29T10:00:00Z"  }}

Lưu ý: signing_url trỏ về một trang HTML cục bộ đơn giản, nơi Developer có thể bấm nút "Simulate Sign" (Giả lập ký) thay vì thực hiện quy trình ký số phức tạp.

Endpoint Giả Lập: POST /electronic-sign/verify

Mục đích: Mô phỏng việc xác thực chữ ký số (CA) theo Nghị định 130.1

Logic Giả Lập: Khi Backend gọi endpoint này với document_id, Mock Server kiểm tra trạng thái trong bộ nhớ tạm (in-memory db). Nếu Developer đã bấm "Simulate Sign", trả về trạng thái hợp lệ.

JSON

{  "code": 0,  "data": {    "status": "COMPLETED",    "signed_hash": "0xabc123...", // Hash giả định để mint token    "verify_result": {      "is_valid": true,      "ca_provider": "Mock-CA-Local",      "timestamp": "2025-12-29T10:05:00Z"    }  }}

### 2.3 Lợi Ích Của Giải Pháp

Bằng cách thay thế FPT.eContract bằng Mock Server này, dự án loại bỏ hoàn toàn chi phí phát sinh cho mỗi lần tạo tài liệu thử nghiệm. Đồng thời, tốc độ phản hồi API giảm từ hàng giây (do xử lý PDF và mạng) xuống mili-giây, tăng tốc độ chạy Integration Test lên đáng kể.

## 3. Lớp 2: Ảo Hóa Thế Giới Vật Lý (IoT & Tuya Ecosystem)

Đây là thách thức lớn nhất khi tái cấu trúc: làm thế nào để kiểm thử logic "Proof of Access" (Bằng chứng truy cập) mà không cần lắp đặt khóa thông minh vật lý? VinaLib-Vault dựa vào nhật ký mở khóa (unlock logs) từ Tuya Cloud để phân chia lợi nhuận.1 Giải pháp là xây dựng một Virtual Device Manager (VDM) - Trình quản lý thiết bị ảo.

### 3.1 Xây Dựng Virtual Device Manager (VDM)

VDM là một service nhỏ chạy song song (hoặc tích hợp trong Mock Server), chịu trách nhiệm duy trì "trạng thái vật lý giả định" của các thiết bị. Thay vì kết nối với phần cứng qua MQTT/Wi-Fi 1, VDM lưu trữ trạng thái trong cơ sở dữ liệu cục bộ (như Redis hoặc file JSON).

Các Chức Năng Cốt Lõi Của VDM:

Quản lý trạng thái: Lưu trữ battery_level (pin), online_status (trực tuyến/ngoại tuyến), và lock_state (đóng/mở).

Giả lập hành vi người dùng: Cung cấp API để Developer kích hoạt sự kiện "Mở cửa bằng vân tay" hoặc "Mở cửa bằng mật khẩu".

Phản hồi API chuẩn Tuya: Trả về dữ liệu đúng định dạng mà SDK Tuya mong đợi.

### 3.2 Mocking Tuya API: Data Plane & Control Plane

Hệ thống cần mock hai nhóm API chính của Tuya: Điều khiển (Control) và Dữ liệu (Data).5

#### Giả Lập Control Plane: Tạo Mật Khẩu Tạm Thời (Temporary Password)

Trong thực tế, việc tạo mật khẩu yêu cầu mã hóa AES-128 và giao tiếp với phần cứng.1 Trên localhost, ta loại bỏ lớp mã hóa này.

Endpoint: POST /v1.0/devices/{device_id}/door-lock/password-free/door-operate

Mock Logic:

Nhận request tạo ticket.

VDM ghi nhận rằng "Một yêu cầu mở khóa đang chờ xử lý".

Trả về JSON thành công ngay lập tức.

JSON

#### Giả Lập Data Plane: Truy Xuất Nhật Ký (Unlock Records)

Đây là dữ liệu quan trọng nhất để tính toán lợi suất (Yield Farming) trong VinaLib-Vault. Chúng ta cần giả lập chính xác cấu trúc JSON của nhật ký Tuya.8

Endpoint: GET /v1.0/devices/{device_id}/logs

Mock Logic: VDM trả về danh sách các sự kiện mở cửa đã được "tiêm" vào hệ thống bởi Developer.

Cấu Trúc JSON Phức Hợp Cho Mocking:

Dựa trên tài liệu Tuya 7, mock payload phải bao gồm code (loại mở khóa) và value (ID người dùng):

Phân tích: Bằng cách mock trường update_time, Developer có thể kiểm thử các kịch bản thời gian (time-travel testing): Điều gì xảy ra nếu khách mở cửa sau khi hết hạn hợp đồng thuê? Hệ thống thật không thể tua ngược thời gian, nhưng Mock Server cho phép điều này dễ dàng.

### 3.3 Bypass Xác Thực API (API Signing)

Tuya API yêu cầu ký HMAC-SHA256 phức tạp với client_id, access_token, t (timestamp).10

Trong môi trường Local, việc tính toán signature này gây lãng phí thời gian và dễ lỗi.

Giải pháp: Cấu hình Backend VinaLib để sử dụng MockIoTAdapter. Adapter này sẽ bỏ qua bước ký (signing) và gửi request trần (plain request) đến Mock Server. Mock Server được cấu hình để chấp nhận mọi request có header Authorization: Bearer mock-token, loại bỏ hoàn toàn rào cản xác thực.

VinaLib-Vault sử dụng Chainlink Functions để kết nối Smart Contract với API bên ngoài.1 Đây là thành phần khó giả lập nhất vì nó liên quan đến việc thực thi mã JavaScript trong một mạng lưới phi tập trung (DON). Tuy nhiên, với sự ra đời của Chainlink Local, chúng ta có thể chạy toàn bộ quy trình này trên máy cá nhân.11

Thay vì kết nối với mạng thử nghiệm (Sepolia/Fuji) và chờ đợi các nút Oracle phản hồi (tốn LINK token testnet), chúng ta sử dụng gói @chainlink/local để triển khai một CCIPLocalSimulator và FunctionsRouterMock ngay trên mạng Hardhat cục bộ.12

### 4.2 Giả Lập Thực Thi Functions (Functions Simulation)

VinaLib-Vault sử dụng mã JavaScript để gọi Tuya API. Để test đoạn mã này mà không cần gửi lên Blockchain, ta sử dụng công cụ simulateScript từ gói @chainlink/functions-toolkit.15

Cấu hình Simulation Script (sim-request.js):

Trong môi trường Production, biến source trỏ đến https://openapi.tuyaus.com. Trong môi trường Local, chúng ta cần kỹ thuật Dynamic URL Injection (Tiêm URL động).

JavaScript

// Mô phỏng logic Javascript của Oracle const { simulateScript } = require("@chainlink/functions-toolkit");const requestConfig = {  source: fs.readFileSync("./API-request.js").toString(),  // Thay thế URL thật bằng URL của Mock Server trên localhost  secrets: { tuyaApiUrl: "http://localhost:3000/v1.0" },   args: ["device-123", "2025-01-01"]};// Chạy mô phỏng cục bộ - Không tốn phí gas, phản hồi tức thìconst { responseBytesHexstring } = await simulateScript(requestConfig);

Chi tiết quan trọng: Trong môi trường thật, Secrets (API Key) được mã hóa (Encrypted). Trong Local, simulateScript cho phép truyền Secrets dưới dạng plaintext.11 Điều này đơn giản hóa quá trình debug cực kỳ lớn. Bằng cách trỏ tuyaApiUrl về http://localhost:3000, mã Oracle JS sẽ gọi đến VDM (đã thiết lập ở Mục 3) thay vì gọi sang Server Trung Quốc của Tuya, đảm bảo độ trễ gần như bằng 0.

### 4.3 Giả Lập Phản Hồi Oracle (Fulfillment Mocking)

Sau khi Smart Contract gửi request, nó chờ callback handleOracleFulfillment. Trên Mainnet, việc này mất vài block. Trên Localhost, ta có thể kích hoạt nó ngay lập tức trong cùng một transaction hoặc transaction kế tiếp.

Sử dụng Hardhat task để gọi hàm fulfillRequest trên FunctionsRouterMock.17

Input: requestId (từ sự kiện Request), response (kết quả từ simulateScript).

Action: Router Mock sẽ gọi lại consumer contract ngay lập tức.

Lợi ích: Kiểm thử được logic xử lý dữ liệu (decoding) và logic nghiệp vụ (phân chia lợi nhuận) một cách đồng bộ (synchronous).

## 5. Lớp 4: Blockchain & Tự Động Hóa (Automation/Keepers)

VinaLib-Vault sử dụng Chainlink Automation để kích hoạt các tác vụ định kỳ (ví dụ: quét trạng thái khóa cửa mỗi 30 phút). Trên Localhost, không có mạng lưới Keepers nào đang chạy để "ping" hợp đồng của bạn.

### 5.1 Giả Lập "Heartbeat" (Nhịp Tim) Của Hệ Thống

Chúng ta thay thế mạng lưới Keepers bằng một Hardhat Script hoặc một tiến trình Cron Job chạy trên máy phát triển.12

Kịch Bản Script (mock-keeper.js):

Check: Gọi hàm checkUpkeep() trên Smart Contract. Đây là hàm view, không tốn gas.19

Verify: Nếu kết quả trả về upkeepNeeded == true.

Execute: Script tự động gửi transaction gọi hàm performUpkeep(performData).18

Tích Hợp Vào Quy Trình Test:

Trong các test case (ví dụ: test kịch bản hết hạn thuê), Developer có thể gọi trực tiếp performUpkeep mà không cần chờ đợi thời gian thực.18 Điều này cho phép "tua nhanh" thời gian bằng cách sử dụng evm_increaseTime của Hardhat Network, kiểm tra xem hợp đồng có xử lý đúng logic quá hạn hay không chỉ trong vài giây thay vì vài ngày.

### 6.2 Xây Dựng "Bank Simulator" (Giả Lập Ngân Hàng)

Developer xây dựng một công cụ nhỏ (ví dụ: /admin/simulate-payment).

Input: Nhập Booking Code và Amount.

Process: Công cụ đóng gói dữ liệu thành JSON theo mẫu trên.

Inject: Gửi POST request đến endpoint webhook của Backend VinaLib-Vault (http://localhost:3000/webhook/casso).

Header: Đính kèm header secure-token khớp với cấu hình local để vượt qua lớp bảo mật.20

Kỹ thuật này cho phép kiểm thử các kịch bản tài chính phức tạp:

Thanh toán thiếu: Gửi amount = 4.000.000 (Hệ thống phải báo lỗi hoặc chờ nộp thêm).

Thanh toán dư: Gửi amount = 6.000.000 (Hệ thống ghi nhận số dư thừa).

Spam Webhook: Gửi cùng một Transaction ID nhiều lần (Hệ thống phải xử lý Idempotency - tính lũy đẳng, không cộng tiền 2 lần).

## 7. Kiến Trúc Phần Mềm: Mẫu Thiết Kế Mock-Adapter

Để hiện thực hóa toàn bộ các giải pháp trên mà không làm "bẩn" mã nguồn Production, chúng ta áp dụng mẫu thiết kế Dependency Injection (Tiêm sự phụ thuộc) triệt để trong Backend.

### 7.1 Interface Segregation (Phân Tách Giao Diện)

Định nghĩa các Interface chung cho các dịch vụ:

ILegalService: Định nghĩa hàm createContract, verifySignature.

IIoTService: Định nghĩa hàm generatePasscode, getLogs.

IFinanceService: Định nghĩa hàm verifyTransaction.

### 7.2 Dynamic Binding (Liên Kết Động)

Trong file cấu hình khởi động ứng dụng (ví dụ: app.module.ts trong NestJS), sử dụng biến môi trường NODE_ENV để quyết định implementation nào được nạp.

Nếu NODE_ENV === 'production': Nạp FptLegalService (Gọi API thật), TuyaIoTService (Gọi Cloud).

Nếu NODE_ENV === 'local': Nạp MockLegalService (Gọi Mock Server), MockIoTService (Gọi VDM).

Cách tiếp cận này đảm bảo logic nghiệp vụ (Business Logic) không thay đổi dù chạy ở môi trường nào. Logic xử lý booking, tính toán lợi nhuận, và tương tác blockchain vẫn giữ nguyên, chỉ có "đầu vào/đầu ra" dữ liệu là được chuyển hướng sang các nguồn giả lập zero-cost.

## 8. Kết Luận Và Lộ Trình Triển Khai

Báo cáo đã trình bày chi tiết phương án tái cấu trúc dự án VinaLib-Vault để vận hành hoàn toàn trên Localhost. Bằng cách áp dụng Chainlink Local cho lớp Oracle, xây dựng Virtual Device Manager cho lớp IoT, và sử dụng Mock Servers cho các dịch vụ pháp lý/tài chính, dự án đạt được các mục tiêu cốt lõi:

Chi phí bằng 0: Loại bỏ phí gas, phí API, và phí chữ ký số.

Vận hành toàn diện: Mọi luồng nghiệp vụ từ Booking -> Payment -> Contract -> Access -> Yield đều hoạt động trơn tru.

Tăng tốc độ phát triển: Phản hồi tức thì, không phụ thuộc vào độ trễ mạng hay thời gian xác thực blockchain.

Khuyến nghị lộ trình:

Tuần 1: Thiết lập môi trường Hardhat với @chainlink/local, viết script deploy FunctionsRouterMock.

Tuần 2: Xây dựng Mock Server (Express) tích hợp VDM cho Tuya và Stub cho FPT/Casso.

Tuần 3: Refactor Backend theo mẫu Mock-Adapter, viết Unit Test cho từng Adapter.

Tuần 4: Tích hợp toàn trình (End-to-End Testing), viết script tự động hóa kịch bản "User Journey" đầy đủ trên Localhost.

Kiến trúc này không chỉ tối ưu hóa chi phí mà còn tạo ra nền tảng vững chắc cho quy trình CI/CD, đảm bảo chất lượng phần mềm cao nhất trước khi triển khai ra môi trường thực tế đầy rủi ro.
